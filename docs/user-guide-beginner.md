# Aegis WAF 小白操作说明

这份文档给第一次使用的人看，尽量不讲代码，只说明“页面上怎么填、每个选项是什么意思、常见场景怎么配”。

适用页面：控制台左侧进入 `防护配置`。

---

## 1. 先理解几个基础概念

### 1.1 WAF 是怎么拦请求的

一个请求进来后，大致会走这条链路：

```text
用户请求
  -> 匹配站点
  -> 访问控制 / 白名单
  -> CC / Bot 防护
  -> 规则检测 / CRS / 自定义规则
  -> 语义检测
  -> 决定 allow / observe / captcha / block
  -> 写访问日志或攻击日志
```

### 1.2 页面里常见动作是什么意思

| 动作 | 含义 | 用户看到什么 |
| --- | --- | --- |
| allow | 放行 | 请求正常转发到后端网站 |
| observe | 观察 | 记录日志，但不拦截 |
| captcha | 人机验证 | 要求通过验证码后再访问 |
| block | 拦截 | 返回 403，不转发到后端 |
| temp-block | 临时封禁 | 一段时间内直接拦截这个来源 |
| long-block | 长期封禁 | 更长时间封禁这个来源 |

## 2. 新增自定义防护规则怎么填

入口：`防护配置` -> `规则管理 / CRS` -> `新增自定义规则`。

保存按钮叫 `保存并热更新`，意思是保存后会立即刷新运行时规则，后续请求会使用新规则。

### 2.1 字段总览

| 字段 | 怎么填 | 例子 | 说明 |
| --- | --- | --- | --- |
| 规则 ID | 填一个正整数，不能和已有规则重复 | `991001` | 规则的唯一编号，建议自定义规则用 900000 以上，避免和 CRS 规则冲突 |
| 名称 | 简短描述这条规则做什么 | `block union select` | 给人看的名字，方便以后搜索 |
| 类别 | 规则属于哪类攻击 | `sqli`、`xss`、`custom` | 会影响规则组统计、站点策略规则组开关和日志展示 |
| 变量 | 检查请求的哪个位置 | `ARGS` | 下拉框，见下面详细说明 |
| 操作符 | 怎么匹配 | `@contains` | 下拉框，见下面详细说明 |
| 匹配内容 | 要匹配的危险字符串 | `union select` | 命中这个内容后触发规则 |
| 处置 | 命中后怎么处理 | `deny` | 下拉框，见下面详细说明 |
| 等级 | 风险等级 | `high` | 影响日志风险展示，也可辅助默认分数判断 |
| 分数 | 命中后加多少异常分 | `7` | 分数达到站点阈值才会拦截；低于阈值通常是 observe |
| 来源 | 规则来源 | `custom` | 自己新增一般选 custom |
| 状态 | 是否启用 | `启用` | 禁用后规则还在，但不参与检测 |
| 描述 | 备注 | `拦截 SQL 注入测试语句` | 写清楚为什么加这条规则，方便回溯 |

## 3. 新增规则：下拉框选项解释

### 3.1 变量：检查请求的哪个位置

| 选项 | 检查哪里 | 适合拦什么 | 示例 |
| --- | --- | --- | --- |
| ARGS | URL 参数和请求参数 | SQLi、XSS、参数里的攻击 payload | `/search?q=union select` |
| REQUEST_URI | 请求地址 | 扫描路径、敏感路径、路径遍历 | `/../../etc/passwd`、`/admin.php` |
| REQUEST_HEADERS | 请求头，目前重点看 User-Agent 和 Content-Type | 恶意 UA、扫描器、工具流量 | `sqlmap`、`acunetix` |
| REQUEST_METHOD | 请求方法 | 禁止危险方法 | `TRACE`、`PUT` |
| BODY/default | 请求体和其它默认内容 | POST body 里的攻击内容 | JSON、表单、原始 body |

小白建议：

- 不知道选哪个，优先选 `ARGS`。
- 拦 URL 路径，选 `REQUEST_URI`。
- 拦扫描器 UA，选 `REQUEST_HEADERS`。
- 拦请求体里的内容，选 `BODY/default`。

### 3.2 操作符：怎么匹配

页面现在有 3 个操作符：

| 选项 | 中文理解 | 匹配方式 | 适合场景 |
| --- | --- | --- | --- |
| @contains | 包含 | 目标内容里只要包含“匹配内容”就命中 | 最常用，推荐新手优先用 |
| @streq | 完全等于 | 目标内容必须和“匹配内容”完全一样才命中 | 拦固定方法、固定路径、固定 UA |
| @rx | 正则/模式匹配入口 | 当前版本运行时按“包含匹配”处理，不是真正完整正则 | 先当成 contains 用；复杂正则后续再增强 |

重要提醒：

- 当前后端 `@rx` 还没有真正做正则引擎匹配，实际效果接近 `@contains`。
- 如果要稳定拦截，优先用 `@contains`。
- `@streq` 很严格，目标内容多一个字符都不会命中。

### 3.3 处置：命中规则后怎么办

| 选项 | 含义 | 实际效果 |
| --- | --- | --- |
| deny | 拦截倾向 | 命中后加分，分数达到站点阈值时 block |
| log | 只记录倾向 | 更适合观察新规则，避免误杀 |
| pass | 放行倾向 | 当前主要用于兼容/保留，不建议新手用来做拦截规则 |

小白建议：

- 新规则不确定会不会误杀：先用 `log` 或者低分数观察。
- 明确是攻击：用 `deny`，分数设到能超过站点阈值。
- 不清楚 `pass` 的用途：不要选。

### 3.4 等级：风险高低

| 选项 | 中文理解 | 建议分数 |
| --- | --- | --- |
| critical | 严重 | 10-15 |
| high | 高危 | 7-10 |
| medium | 中危 | 4-6 |
| low | 低危 | 1-3 |
| info | 信息 | 1 |

说明：

- 等级主要用于展示风险和辅助判断。
- 真正决定是否拦截的是：规则分数 + 站点阈值。
- 例如站点阈值是 `5`，规则分数是 `6`，命中后通常会 block。
- 如果规则分数是 `3`，低于阈值 `5`，通常会 observe，只记日志不拦截。

### 3.5 来源：规则从哪里来

| 选项 | 含义 | 新手怎么选 |
| --- | --- | --- |
| custom | 自定义规则 | 手动新增规则选这个 |
| semantic | 语义规则 | 由语义指纹升级来的规则，一般不用手填 |
| system | 系统规则 | 系统内置规则，一般只读 |
| crs | OWASP CRS 规则 | CRS 规则，一般只读，不建议手动新增为 crs |

小白建议：新增自定义规则时固定选 `custom`。

## 4. 新增规则示例

### 4.1 拦截 SQL 注入：union select

适合场景：有人在参数里尝试 SQL 注入。

| 字段 | 填写 |
| --- | --- |
| 规则 ID | `991001` |
| 名称 | `block union select` |
| 类别 | `sqli` |
| 变量 | `ARGS` |
| 操作符 | `@contains` |
| 匹配内容 | `union select` |
| 处置 | `deny` |
| 等级 | `high` |
| 分数 | `8` |
| 来源 | `custom` |
| 状态 | `启用` |
| 描述 | `拦截参数中的 SQL 注入 union select` |

测试请求：

```text
/search?q=1 union select password from users
```

预期：攻击日志里能看到规则命中，分数达到阈值时返回 403。

### 4.2 拦截 XSS：script 标签

| 字段 | 填写 |
| --- | --- |
| 规则 ID | `991002` |
| 名称 | `block script tag` |
| 类别 | `xss` |
| 变量 | `ARGS` |
| 操作符 | `@contains` |
| 匹配内容 | `<script` |
| 处置 | `deny` |
| 等级 | `high` |
| 分数 | `8` |
| 来源 | `custom` |
| 状态 | `启用` |

测试请求：

```text
/comment?text=<script>alert(1)</script>
```

### 4.3 拦截扫描器 UA：sqlmap

| 字段 | 填写 |
| --- | --- |
| 规则 ID | `991003` |
| 名称 | `block sqlmap ua` |
| 类别 | `scanner` |
| 变量 | `REQUEST_HEADERS` |
| 操作符 | `@contains` |
| 匹配内容 | `sqlmap` |
| 处置 | `deny` |
| 等级 | `medium` |
| 分数 | `6` |
| 来源 | `custom` |
| 状态 | `启用` |

说明：这个规则会检查 User-Agent / Content-Type 这类请求头。

### 4.4 拦截 TRACE 方法

| 字段 | 填写 |
| --- | --- |
| 规则 ID | `991004` |
| 名称 | `block trace method` |
| 类别 | `protocol` |
| 变量 | `REQUEST_METHOD` |
| 操作符 | `@streq` |
| 匹配内容 | `TRACE` |
| 处置 | `deny` |
| 等级 | `medium` |
| 分数 | `6` |
| 来源 | `custom` |
| 状态 | `启用` |

说明：这里用 `@streq`，因为请求方法应该完全等于 `TRACE` 才拦。

## 5. 为什么规则命中了但没有拦截

最常见原因是：命中分数没有达到站点阈值。

例子：

```text
站点拦截阈值 = 5
规则分数 = 3
结果 = observe，只记录日志，不拦截
```

如果你希望命中后直接拦截，可以：

1. 把规则分数调高，比如从 `3` 改成 `6`。
2. 或者把站点策略里的拦截阈值调低。
3. 或者确认规则类别所在的规则组已经启用。

## 6. 规则组和类别怎么理解

类别字段常见值：

| 类别 | 含义 |
| --- | --- |
| sqli | SQL 注入 |
| xss | 跨站脚本 |
| rce | 远程命令执行 |
| lfi | 文件包含 / 路径遍历 |
| scanner | 扫描器 / 探测器 |
| bot | Bot / 自动化流量 |
| protocol | 协议异常 |
| custom | 自定义通用规则 |
| semantic | 语义指纹规则 |

注意：站点策略里可以启停规则组。如果一个站点没有启用某个规则组，这个类别的规则可能不会参与评分。

小白建议：

- SQL 注入填 `sqli`。
- XSS 填 `xss`。
- 扫描器填 `scanner`。
- 不知道填什么就填 `custom`。

## 7. 新增白名单 / 误报例外怎么填

入口：`防护配置` -> `误报白名单` -> `新增白名单/例外`。

适合场景：某个正常业务请求被 WAF 拦了，确认安全后给它加白。

### 7.1 类型选项

| 类型 | 作用 | 示例 |
| --- | --- | --- |
| URL 白名单 | 放行某个 URL 路径 | `/callback` |
| 参数白名单 | 放行某个参数值 | `token=abc` 或参数名 |
| IP 白名单 | 放行单个 IP | `203.0.113.10` |
| CIDR 白名单 | 放行一个网段 | `203.0.113.0/24` |
| Header 白名单 | 放行某个请求头特征 | `User-Agent=InternalBot` |
| Cookie 白名单 | 放行某个 Cookie 特征 | `session=trusted` |
| CRS Rule Exclusion | 排除某条 CRS/规则 ID | `942100` |

### 7.2 作用域选项

| 作用域 | 含义 | 影响范围 |
| --- | --- | --- |
| global | 全局 | 所有站点都生效，慎用 |
| site | 当前站点 | 只影响指定站点，推荐默认用这个 |
| path | 指定路径 | 只影响某个 URL 路径 |
| ruleId | 指定规则 | 只排除某条规则 |
| variable | 指定变量 | 只排除某个参数/请求字段 |

小白建议：

- 能用 `site` 就不要用 `global`。
- 能用 `path` 就不要放大到整个站点。
- 加白一定写原因和过期时间。

### 7.3 字段怎么填

| 字段 | 怎么填 | 说明 |
| --- | --- | --- |
| 站点 ID | 当前站点 ID，空或 0 表示全局 | 新手不要轻易填 0 |
| 类型 | 选择 URL/IP/参数/规则例外等 | 根据误报类型选 |
| 作用域 | site/path/ruleId/variable | 越精确越安全 |
| 匹配值 | 要加白的路径、IP、参数或规则 ID | 例如 `/callback` |
| 规则 ID | 做 rule exclusion 时填写 | 例如 `942100` |
| 变量 | 限定某个变量 | 例如 `ARGS:q` |
| 过期时间 | 可选 | 临时放行建议必须填 |
| 状态 | 启用/禁用 | 禁用后不生效 |
| 原因 | 为什么加白 | 写误报原因或工单号 |

## 8. 从攻击日志里处理误报

入口：`攻击事件` -> 打开某条日志详情。

详情里会展示：

- 命中了哪条规则。
- 为什么拦截。
- 分数是多少，阈值是多少。
- 原始 payload 和规范化后的 payload。
- 白名单 / CC / 语义决策。
- 运营建议。

如果确认是误报，可以点日志里的建议操作，例如：

| 建议 | 含义 |
| --- | --- |
| 生成白名单/例外 | 按当前路径、规则、变量预填白名单 |
| 复核站点策略 | 去看站点阈值和规则组是否太严格 |
| 复核规则组 | 去规则管理看命中的规则类别 |
| 查看 CC/Bot 策略 | 如果是频率/封禁导致，去 CC/Bot 页面处理 |

## 9. 常见问题

### 9.1 我新增规则后为什么没生效？

按顺序检查：

1. 规则状态是不是 `启用`。
2. 来源是不是 `custom`。
3. 规则类别所在的规则组，站点策略是否启用。
4. 变量选对了吗：参数用 `ARGS`，路径用 `REQUEST_URI`，UA 用 `REQUEST_HEADERS`。
5. 分数是否达到站点拦截阈值。
6. 是否被白名单或 rule exclusion 放行。

### 9.2 我应该用 @contains 还是 @streq？

- 大多数情况用 `@contains`。
- 只有你要匹配“完全等于”时才用 `@streq`。
- 例如拦 `TRACE` 方法，用 `REQUEST_METHOD + @streq + TRACE`。

### 9.3 分数应该填多少？

简单建议：

| 确定程度 | 建议分数 |
| --- | --- |
| 100% 是攻击 | 8-10 |
| 大概率是攻击 | 6-8 |
| 不确定，先观察 | 1-4 |

如果站点阈值是 `5`：

- 分数 `6` 通常会拦截。
- 分数 `3` 通常只观察。

### 9.4 白名单是不是越多越好？

不是。白名单会降低防护强度。

建议：

- 白名单范围越小越好。
- 尽量限定站点、路径、规则 ID、变量。
- 临时处理一定设置过期时间。
- 原因写清楚，方便以后清理。

### 9.5 CRS/system 规则能不能直接编辑？

页面提示里写了：CRS/system 运行时规则主要是只读展示。需要修改时，建议克隆或新增一条 `custom` 规则，或者用 `CRS Rule Exclusion` 做精确例外。

## 10. 推荐新手流程

### 10.1 想新增拦截规则

```text
防护配置
  -> 规则管理 / CRS
  -> 新增自定义规则
  -> 来源选 custom
  -> 操作符优先选 @contains
  -> 分数先按风险设置
  -> 保存并热更新
  -> 用测试请求验证
  -> 看攻击日志解释
```

### 10.2 想处理误报

```text
攻击事件
  -> 打开详情
  -> 看命中规则和变量
  -> 确认是正常业务
  -> 使用运营建议生成白名单/例外
  -> 限定 site/path/ruleId/variable
  -> 设置过期时间和原因
  -> 再次请求验证是否放行
```

### 10.3 想先观察不拦截

```text
新增规则
  -> 处置选 log 或分数设低
  -> 保存
  -> 观察攻击日志
  -> 确认没有误报后再提高分数或改 deny
```

## 11. 当前版本要特别注意的限制

1. `@rx` 当前运行时效果接近 `@contains`，还不是真正完整正则匹配。
2. 自定义规则主要适合明确字符串、路径、UA、方法、参数等场景。
3. 复杂业务逻辑漏洞、越权、支付风控这类问题不能只靠 WAF 规则解决。
4. 白名单会降低防护强度，必须小范围、可过期、写原因。
5. 生产上线前还需要做全量回归、压测、真实流量试运行和误报率评估。
